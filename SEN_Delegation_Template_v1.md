# SEN Delegation Template

## モデル非依存・依頼文テンプレート（共通層）

> **目的**：実装作業を、特定の AI モデルに依存せず委譲するための器。
> **接続**：第3部 第5章「抽象と具体の役割分担（依頼文の哲学）」＋
> 第11章「LLM いいとこ取り OS」11.3「共通インターフェース＋モデル別アダプタ」。
> **思想**：人間は抽象（何を・なぜ）を書く。モデルは具体（どう）を実装する。
> この依頼文は **共通層**である。Codex / Claude / ローカル LLM のいずれに渡しても、
> 同じ意図で実行できることを目指す。

---

## このテンプレートの位置づけ

これは「タスクの抽象定義」である。
**モデルを知らない**ように書く——
特定モデルのプロンプト作法・ツール名・口調を埋め込まない。

モデル固有の最適化（プロンプト整形、ツール呼び出し、出力規約）は、
**アダプタ層**——つまり各モデルに渡す直前の薄い変換——で吸収する（11.3）。

ゆえに、このテンプレートを一度書けば、
モデルを乗り換えても器は残る（第25条「モデルは借り物」）。

---

## テンプレート本体（コピーして使う）

```markdown
# 依頼：<タスク名>

## 0. メタ
- 依頼者：<人間の名前 / 役割>
- 実行者：<任意の LLM（Codex / Claude / ローカル等）>
- このタスクの抽象度：<実装 / 調査 / 整理 / 設計 のいずれか>
- SEN Design 準拠：はい（共通層として記述）

## 1. 目的（なぜ・何を）
<一文で。世の中／プロジェクトの何の必要に応えるのか（第1条）。>
例）能力バンクに「調査キャプチャ」を汎用の能力として追加し、
   モデルを乗り換えても再利用できる形で蓄積できるようにする。

## 2. 文脈（能力バンクから注入）
- 対象リポジトリ / Vault：<パス。例：~/vaults/sen-...>
- 既存の構造：<関連するディレクトリ・ファイル・命名規則>
- 前提・制約：<守るべき規約、依存関係、触ってはいけない箇所>
- 参照すべき資産：<過去の手順・調査キャプチャ・設計メモ>

## 3. 成果物（期待するアウトプット）
- <作成 / 変更するファイルと、その役割>
- <成果物の形式（Markdown / コード / 設定ファイル等）>
- <最小単位で完結すること。1依頼1目的。>

## 4. 制約・原則（SEN Design）
- データ主権（第5条）：機密・個人情報は外部に出さない。境界線の内側で処理。
- 摩擦配置（第4条）：
  - 自動でよい：閲覧・調査・新規ファイル作成
  - 確認を取る：既存ファイル編集・依存追加・コミット
  - 必ず止まる：削除・リモートへの破壊的操作・外部 API 書き込み・課金
- 抽象を侵さない（第7条）：設計判断・哲学・優先順位は実行者が勝手に変えない。
  迷ったら**実装を止めて問いを返す**。

## 5. 評価基準（完了の定義 / ルーブリック）
- [ ] 目的（1.）を満たしているか
- [ ] 既存の構造・命名規則に整合しているか
- [ ] モデル固有の前提に依存していないか（次のモデルでも再利用可能か）
- [ ] <タスク固有の合否条件を 1〜3 個>

## 6. 禁止事項
- 指示にない範囲への波及的な変更（大規模リファクタは事前確認）
- 文脈（2.）に無い前提の捏造
- 評価基準（5.）を満たさないままの「完了」報告

## 7. 引き継ぎ（出力フォーマット）
- 変更点の要約（何を・なぜ）
- 評価基準（5.）の各項目に対する自己チェック結果
- 判断に迷い、勝手に決めた箇所があれば明示（後で人間が視差を当てる）
- 次の一手の提案（あれば）
```

## 記入例（スクショの別 Vault 作業を共通層で書くと）

```markdown
# 依頼：能力バンクに「調査キャプチャ」を汎用能力として追加

## 0. メタ
- 依頼者：人間（抽象側）
- 実行者：任意の LLM（今回は Codex を想定。Claude / ローカルでも可）
- 抽象度：実装
- SEN Design 準拠：はい

## 1. 目的
LLM ごとの調査結果を、モデルの外側に蓄積する「能力バンク」を作り、
「調査キャプチャ」をその第一の能力として登録する。
モデルを乗り換えても、過去の調査を再利用できる状態にする（第25条）。

## 2. 文脈
- 対象 Vault：~/vaults/sen-...（実パスは依頼時に確定）
- 既存の構造：Obsidian Vault。能力は Markdown ノートとして管理。
- 前提：中断前のローカル変更は Vault 未反映。まず差分を確認してから着手。
- 参照資産：過去の調査ノート、既存の能力ノートの命名規則。

## 3. 成果物
- 能力バンクの器となるディレクトリ／インデックスノート
- 「調査キャプチャ」能力ノート（入力・手順・評価基準を含む）
- 既存の調査結果を取り込むための雛形

## 4. 制約・原則
- データ主権：Vault はローカル。外部 API に Vault 内容を送らない。
- 摩擦配置：新規ノート作成は自動可。既存ノート編集は差分提示の上で確認。
- 抽象を侵さない：能力の分類体系・命名は人間の設計に従う。迷ったら問いを返す。

## 5. 評価基準
- [ ] 能力バンクに「調査キャプチャ」が独立した能力として登録されている
- [ ] 次に別モデルが来ても、同じノートを読んで調査を再現できる
- [ ] 既存の Vault 命名規則・構造に整合している

## 6. 禁止事項
- Vault 全体の再編成（インデックス追加に留める）
- 中断前ローカル変更の上書き（先に差分確認）

## 7. 引き継ぎ
- 作成／変更したノートの一覧と役割
- 評価基準の自己チェック結果
- 設計上、勝手に決めた分類があれば明示
```

## 使い方

1. **共通層を書く**：上のテンプレートを、人間（抽象側）が記入する。
   モデルを意識せず、「何を・なぜ・どこまで」だけを書く。
2. **アダプタを当てる**：渡す相手に応じて、薄い変換を足す（11.3）。
   - Codex 向け：実行環境・対象パス・ツール権限を明示
   - Claude 向け：文脈の言語化を厚めに、対話で詰める前提
   - ローカル LLM 向け：タスクを小さく分割、評価基準を機械可読に
3. **評価で磨く**：完了後、評価基準（5.）の充足度を記録する。
   どのモデル／アダプタが、どのタスクで良かったかを残す（11.5）。
   この記録が、次回のルーティング判断を改善する。

## なぜ「共通層」を人間が書くのか

依頼文は、哲学を実装に翻訳する経路である（第5章）。

「どう実装するか（具体）」はモデルに委ねてよい。
しかし「何を・なぜ・どこまで（抽象）」は、人間が握る。
ここを手放した瞬間、レバレッジが消え、暴走が始まる（第7条・第17条）。

モデルは借り物である（第25条）。
借り物に渡すのは具体の実行であって、
目的と評価基準——器そのもの——ではない。

追うのではなく、器を持つ。
この依頼文が、その器の一枚である。
